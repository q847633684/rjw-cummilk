# 代码建议与扩展方向

基于当前 rjw-cummilk 的泌乳系统设计原则、文档与 Cumpilation 合并结构，整理**建议**（架构/质量/文档）与**扩展方向**（功能与集成），供后续迭代参考。

---

## 一、建议（架构与质量）

### 1.1 继续收束「只读游戏」

- **耐受增量**：按《游戏已接管变量与机制清单》第三节，若确认原版在服用带 chemical 的药物时**已自动**按 ChemicalDef 增加 toleranceHediff 严重度，可尝试在 outcomeDoers 中**去掉**单独 `GiveHediff(EM_Prolactin_Tolerance)`，只保留 HediffDef 的 SeverityPerDay 做自然恢复，并在小版本里验证行为一致。
- **E_tol 单点**：已收束到 `GetProlactinToleranceFactor(pawn)`；新增需求（如 UI 显示、平衡表）时只从此处或 Def 读，避免再散落手写公式。
- **成瘾/戒断心情**：文档建议戒断/满足优先用原版化学心情；若当前 EM_* Thought 与原版重复，可逐步改为挂接原版或仅补差，减少重复 Def。

### 1.2 泌乳与 Cumpilation 的边界

- **共用层**：`CompShowProducer` + allowedConsumers/allowedSucklers 已统一「产主 + 谁可吃」；建议在文档中明确「奶与精液共用同一套消费者规则」，避免后续在精液或奶一侧单独加例外逻辑。
- **设置入口**：`EqualMilkingSettings` 同时承载泌乳与 Cumpilation 子标签；若 Cumpilation 选项继续增多，可考虑按命名空间或子类拆分（如 `CumpilationSettings` 只读/写自己的 Scribe），主 Mod 设置只做聚合与 Tab，便于多人协作和查找。
- **CumpilationIntegration**：当前用 `AccessTools.TypeByName` 按名查找 Cumpilation 类型再 Patch，已避免强依赖；若 Cumpilation 内部重命名，需同步更新字符串。可考虑在 Cumpilation 内暴露少量静态入口（如 `CumpilationIntegration.ApplyFromCumpilation(harmony)`），由 Cumpilation 自己注册需要「产主/消费者」的 hook，减少主程序集对类名的依赖。

### 1.3 文档与可维护性

- **设计原则落地**：《与游戏本体及RJW的冲突与优化总结》中的「设计/落地原则」与「实现说明」已写；建议在 **README 或 About 的 mod 描述** 中加一句「泌乳逻辑遵循只读原版、只挂接不改写」的简短说明，方便贡献者与玩家理解定位。
- **报错修复记录**：继续沿用《报错修复记录》格式；若某类错误重复出现，在「重复修复·根因分析」里补一段，并考虑在代码里加注释指向该文档条目（如 `// 见 Docs/报错修复记录 某条`）。
- **Cumpilation 文档**：若后续主要维护泌乳侧，建议在 Docs 下增加一份 **Cumpilation 模块清单**（类似「除泌乳外还有哪些」的简表 + 关键类/入口），便于快速定位放气、收集、桶、心情等逻辑，避免误改。

### 1.4 测试与兼容

- **RJW 版本**：`RJWVersionDiffHelper` 与 `SexFluidDef` 存在性检查已做；RJW 大版本升级时建议在测试清单里加一项「RJW 分娩 + 泌乳水池」「RJW 乳房倍率 + 挤奶」的回归。
- **无 RJW 路径**：关键路径（如 `GetBreastList`、RJW 专用 patch）已有 null/条件判断；可再梳理一遍「未装 RJW 时是否只退化为对称双池、无 RJW 专用 UI」，避免漏判导致 NRE。
- **PipeSystem 可选**：条件引用与 `PipeSystemPath` 已支持；若后续增加更多「可选 mod」联动，可统一用「OptionalModIntegration」风格（按类型存在再 Patch、再引用），便于维护。

## 二、扩展方向（功能与集成）

### 2.1 泌乳侧

| 方向 | 说明 |
|------|------|
| **药物/成瘾信息面板** | 当前「药物浓度/有效浓度时间」与池模型不一致（文档 6.2 已说明）；若希望玩家更直观，可考虑在**物品信息或健康页**增加一句 Keyed 说明（如「实际持续时间以泌乳池为准」），或提供 Mod 设置中的「基准持续天数」简要说明，减少困惑。 |
| **多剂量/叠加** | 已有 L 叠加与 severity 合并；若需「多次服药在 UI 上更清晰」，可在悬停或殖民者列中增加「当前剂量层数/等效剂量」的只读显示，数据仍来自现有 L 与公式。 |
| **机械体/非人类** | 已有机械体泌乳能耗与 `MechEnergyUsageFactor`；若社区有需求，可扩展为「机械体产奶类型/产物 Def」可配置（仍用 namesToProducts 或等价开关层，不重写底层）。 |
| **剧情/成就** | 在现有 Thought/Memory 基础上，可增加少量「首次泌乳」「长期药物泌乳」等 HistoryEvent 或成就类记忆，挂接原版事件系统，不重复造轮子。 |
| **平衡与 Mod 设置** | 四层模型参数（压力、反射、炎症、激素饱和、组织适应、质量）已可调；可把《参数联动表》中常用组合做成 **预设**（如「更宽松 / 更真实」），一键切换，便于不同口味玩家。 |

### 2.2 Cumpilation 侧

| 方向 | 说明 |
|------|------|
| **桶/产主规则** | 当前单源设 producer、混合桶不设；可文档化「何时设、何时不设」的规则，并在 CumpilationIntegration 注释中写明，便于后续扩展（如「多人贡献同一桶」的标签策略）。 |
| **心情与记录** | Reactions 已有食用记录与阶段心情；若需更细粒度（如按流体类型、来源），可在现有 Fluid_Record_Mapping 上扩展，仍用「只读游戏/只挂接」思路。 |
| **管道与精液** | 若 PipeSystem 已支持奶进管道，可评估「精液是否也进同一管道/同一资源类型」的玩法与平衡，若可行则作为可选联动，与现有 PipeSystem 条件引用一致。 |
| **Slug / Biotech** | Slug 相关已有一层；Biotech 更新若有新机制（如新种族、新流体），可预留「按 Def 或种族 tag 决定是否走 Cumpilation 逻辑」的扩展点，避免硬编码种族名。 |

### 2.3 共用层

| 方向 | 说明 |
|------|------|
| **生产者–消费者 UI** | 当前 allowedConsumers/allowedSucklers 在设置与殖民者列中可见；可考虑在**健康页或物品悬浮**显示「产主」与「谁被允许使用」（仅读现有 Comp 数据），减少重复点进设置。 |
| **统一「流体」抽象** | 奶与精液在「产主 + 允许消费者」上已统一；若未来有第三种流体（如其它 mod 的产物），可考虑抽象一层「IFluidProducer / IConsumableBy」接口，由奶/精液各实现，便于第三方 mod 挂接。 |
| **性能** | 双池、四层模型每 30 tick 更新，规模大时若有性能诉求，可对「非泌乳 pawn」早退、或对池更新做 LOD（如不在当前地图的 pawn 降低更新频率），需结合 Profiler 验证。 |

### 2.4 RJW / 第三方

| 方向 | 说明 |
|------|------|
| **RJW 统一接口** | 文档已提「若 RJW 提供统一可挤奶/产奶接口可优先使用」；若 RJW 未来提供，可收敛 `GetBreastList`、`fluidMultiplier` 等调用到单一 facade，减少对 RJW 内部结构的依赖。 |
| **rjw-genes** | 基因倍率已集成；若有新基因（如新体型、新产奶类型），可保持「只读基因 Def + 倍率」的扩展方式，避免本 mod 反写基因逻辑。 |
| **DBH / 卫生** | 已有 `DubsBadHygieneIntegration`；乳腺炎与卫生的联动若需加强，可在现有卫生系数上扩展（仍读 DBH 数据），不重复实现卫生系统。 |
| **兼容列表** | About 中已列 incompatibleWith；建议在 Docs 中维护一份「已知兼容/不兼容 mod」简短列表（含现象与规避方式），随社区反馈更新。 |

---

## 三、小结

- **建议**：继续收束耐受/成瘾为「只读游戏」、明确奶与精液共用规则、文档化 Cumpilation 模块与设计原则、巩固报错记录与 RJW/可选 mod 的兼容策略。
- **扩展**：泌乳侧可做信息与预设、机械体/剧情小扩展；Cumpilation 侧可文档化产主规则、心情与管道；共用层可做生产者–消费者 UI 与（可选）流体抽象；RJW/第三方保持「只挂接、用其接口」的扩展方向。

以上均可在不违背现有「只读、只挂接」原则的前提下，按优先级逐步落地。
