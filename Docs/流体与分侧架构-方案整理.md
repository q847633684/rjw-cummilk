# 流体与分侧架构 — 方案整理（对话共识）

本文档整理自「刻意忘掉泌乳/妹汁/精液/肛门各自为政的池子，只站在 RJW 编辑器可见量上重想」起的讨论，至**当前已定决策**：**暂不押真实左右身体部位，采用虚拟分侧；待 RJW 更新或自研完整解剖后再接真实 Part。**

---

## 1. 背景与问题

- 现有 Equal Milking / Cumpilation 等多套字典与倍率链并行，复杂度高，易出现**同源变量被容量与流速重复放大**。
- RJW 角色编辑器暴露：**尺寸（严重度/阶段）**、**体液乘率**（`partFluidMultiplier`）、**体液类型**（`SexFluidDef`）、**计算后的体液量与液体倍率**（`FluidAmount` / `GetFluidMultiplier`）。
- 目标：判断**是否大改**，以及**左/右乳、阴囊/附睾/精囊**等分侧玩法如何落地。

---

## 2. RJW「体液」在代码里的含义（编辑器与实现对齐）

| 概念 | 对应实现 | 说明 |
|------|----------|------|
| 体液类型 | `SexFluidDef`，可 `fluidOverride` | 奶、精、妹汁等 |
| 体液乘率（滑条） | `HediffComp_SexPart.partFluidMultiplier` | 个体长期倍率，存档 |
| 液体倍率 | `GetFluidMultiplier()` | 含 Def 基数、年龄、体型、严重度、`partFluidMultiplier` 等 |
| 体液量显示 | `FluidAmount = BaseFluidAmount(20) × GetFluidMultiplier()` | 与倍率**线性同源**，差常数 20 |

**结论**：`FluidAmount` 与 `GetFluidMultiplier` 不要当两套独立信息同时写入容量与流速，避免重复计严重度/体型。

**建议用法**（讨论共识）：

- **流速 / 进水 / 高潮产液倍率**：以 **`GetFluidMultiplier`（或等价 `FluidAmount/20`）** 为主。
- **容量 / 满度上限**：以 **解剖体量**（如 RJW 胸体积/重量或严重度档**择一**作主因子）为主，**避免**再用同一套乘子把容量与流速各乘一遍。

---

## 3. 生物学三层抽象（跨奶/汁/精共用语言）

1. **分泌能力**：单位时间产多少、反射多强 → 游戏上主轴对齐 RJW **倍率链**。
2. **可蓄积腔室**：腺泡、导管、精囊等「暂存」→ 游戏上用 **相对充盈 `fill01`（0～1）** 或等价标度，**上限**与器官体量挂钩，**不**与 RJW 倍率二次平方。
3. **排空事件**：挤奶、射精、吸吮 → 扣减充盈、生成物品/给对方效果。

**妹汁 / 阴道润滑**（讨论共识）：更贴近 **前庭大腺（Bartholin）与充血渗出**，生理上**非大容量水箱**；游戏上适合 **兴奋驱动、薄层/低上限、快平衡**，**不必**与乳房同构的「大池」。

---

## 4. 偏大改方向：`PartFluidState` 服务（目标架构）

- **所有「池」读写收口**到单一服务（或单一存档表），键为稳定 **`FluidSiteKey`**（建议：`Pawn` + 侧别/槽位 + 流体族枚举；未来可绑定 `BodyPartRecord`）。
- **RJW 只读**：`SexFluidDef`、`GetFluidMultiplier()`、性征挂在哪个 Part（将来真实解剖时）。
- **删除/合并**：多套 `Dictionary<string,float>` 与重复倍率链；旧档兼容在本次决策中**不要求**（可破坏性迁移）。

**每个 Site 最小字段（建议）**：

- `fill01`：相对充盈。
- 可选：`letdown` / 冷却（射乳反射、吸奶侧化）。
- 可选：炎症标量（若仍要乳腺炎侧化）。

**API 示意**：`GetFill` / `AddFill` / `TryDrain(partOrSlot, …)` / `Tick`；注册随性征 `PostAdd`，缺失随 `PostRemove`/`MissingPart`。

---

## 5. 左/右乳：理想锚点 vs 当前游戏现实

**理想（长期）**：

- **左乳池 / 右乳池** 押在 **两个 `BodyPartRecord`（同一 `Breast` Def + 不同 `customLabel`）** 上；每条 **乳房 Hediff 的 `Part`** 指向对应叶。
- 吸奶/挤奶 **必须带侧**：`TryDrain` 只扣该侧 `fill01`；可选 let-down 提高该侧有效效率。

**RJW 默认**：`defaultBodyPart` 等为 **Chest**，编辑器里常见 **「一对乳房」挂在胸上**；与「左右叶」不一致时，健康与部位列表**不会出现**左/右子节点。

**兼容性（讨论结论）**：

- **单一 `Breast` Def + 两条身体记录**（非 `LeftBreast`/`RightBreast` 两个 Def）是 **最小侵入**，类比左右肾/肺。
- 仍可能需 Harmony **补 `get_breastsBPR` / `add_breasts` / `GetBreastList`** 与**手术锚点**补丁；**后加载 mod 覆盖 `Human` Chest** 会冲掉左右叶，需排序与测试。
- **另 AI 提议的 `LeftBreast`+`RightBreast` 两个 BodyPartDef** 会放大与 RJW `breastsDef == Chest`、手术、第三方身体 mod 的冲突面，**不推荐**作为首选。

## 6. 精液：当前 MilkCum 实现摘要（代码审计）

- **无**捐赠方长期「睾丸精池」字典；射出量来自 RJW 当时 **`HediffComp_SexPart.FluidAmount`**。
- **接受方**：`SexUtility.TransferFluids` 的 Postfix → **Cumflation**（阴道向：`Cumpilation_Cumflation` severity，标度 `fluidAmount / (100×BodySize×modifier)`）+ **`HediffComp_SourceStorage`**；**Stuffing**（口/肛等，按体液类型 Hediff + 上限/溢出）。
- **泄漏**：`HediffComp_LeakCum` 等扣 severity、按来源产 filth；另有收集/手术/桶与 **producer** 集成。

若未来要做 **左/右睾丸（附睾+精囊）虚拟或真实池**，需在 **射精前扣减** 等路径**新增**逻辑（RJW 默认不持久化精囊存量）。

---

## 7. 用户倾向的生物学映射（讨论中的取向）

- **需要「池」感**：双侧乳房泌乳、双侧睾丸系储精（游戏可简化为「侧槽」或「阴囊+双附睾+精囊」抽象）。
- **不需要大池**：妹汁 / 阴道润滑 → 兴奋驱动、Bartholin 叙事、小容量或纯状态。

（与第 3 节一致，可作为实现优先级分层依据。）

---

## 8. UI / 身体列表预期（RimWorld 规则）

- 健康页树是 **部位树 + 每部位挂 Hediff**，不是「人类乳房 ▶ 左乳」这种 Hediff 套娃。
- 正确形态接近：**胸 → 左乳 / 右乳（子部位）→ 各挂乳房 Hediff**；若只有 **胸 → 一对乳房**，则说明 **身体图仍为 Chest 单锚点** 或 Hediff 未挂到叶子上。

---

## 9. 已定决策（当前阶段）

**放弃以「真实左右身体部位」作为池键的唯一来源；全部采用虚拟分侧（如 `LeftBreast`/`RightBreast` 枚举槽位 + `Pawn` 级状态服务）。**

- **动机**：游戏中 RJW/加载顺序/旧小人常仍表现为 **Chest 单点**，真实 Part 不稳定；维护成本与兼容风险在 RJW 未官方化分叶前偏高。
- **后续**：若 **RJW 更新**提供稳定分侧，或本 mod **另做一套完整真实部位**并通过补丁与手术全面对齐，再 **把 `FluidSiteKey` 从虚拟槽迁移到 `BodyPartRecord`**（或双轨读：有叶用 Part，无叶用槽）。

---

## 10. 建议实施顺序（虚拟侧先行）

1. 定义 **`FluidSiteKey`**：枚举槽位（如 `BreastLeft` / `BreastRight` / 未来 `TesticleLeft`…）+ `Pawn`，**不**依赖 `BodyPartRecord`。
2. 实现 **`PartFluidState`**（或更名为 `FluidSiteState`）单表读写 + Tick + `TryDrain`。
3. **泌乳**：从现有 `poolKey` 逻辑迁到虚拟左/右槽；UI/Job 显式选侧。
4. **精液储备**（若要做）：虚拟左/右睾丸槽 + 与 `TransferFluids` 或自定义射精钩子同步扣增。
5. **妹汁**：状态机或小标量，**不**进大池表。
6. 日后 **真实 Part 可用** 时：加一层 **解析器** `ResolveSiteKey(pawn) → BodyPartRecord?`，有则校验缺失与手术，无则回退虚拟槽。
