# 逻辑审阅：营养与泌乳一致性

- **type**: design
- **date**: 2025-03
- **tags**: #审阅 #营养 #乳池 #1:1 #Need_Food
- **related_docs**: [泌乳系统逻辑图](../docs/泌乳系统逻辑图.md) 第一、五、六节

## 审阅范围

在「营养↔乳池 1:1、Need_Food 直接扣/加饱食度」方案下，对以下链路做一致性检查：

1. 营养→乳池（产奶扣饱食度）
2. 乳池→营养（吸奶/挤奶消费、回缩加回饱食度）
3. 吸收延迟与药物进水
4. 边界与可读性

---

## 1. 营养→乳池（产奶扣饱食度）

**位置**：`MilkingPatch.Need_Food_Patch.NeedInterval_Postfix`

- 条件：`IsMilkable`、有 `CompEquallyMilkable`、有 Lactating 且带 `HediffComp_EqualMilkingLactating`。
- 未满池：`extraFall = flowPerDay * factor * (150/60000)`，从 `CurLevel` 扣除，与规格「1 池单位 = 1 营养」一致；`lactationExtraNutritionBasis`（0–300）映射为 factor（0–2），basis=150 时每日扣减 ≈ flowPerDay。
- 满池且回缩：`addBack = reabsorbedPerDay * factor * intervalPerDay`，加回饱食度；与 `CompEquallyMilkable.GetReabsorbedNutritionPerDay()`（已乘 `NutritionPerPoolUnit` 与效率）一致。

**结论**：与设计一致，扣/加均受同一 factor 控制。

---

## 2. 乳池→营养（吸奶/挤奶）

**GreedyConsume（Charge 消费）**：`BreastfeedPatch.HediffComp_Chargeable_Patch.GreedyConsume_Prefix`

- 从 `CompEquallyMilkable` 用 `DrainForConsume(num, drainedKeys)` 扣池，`num = Min(desiredCharge, Charge)`，返回 `__result = num`；未再做池↔营养换算，1:1。

**直接吸奶**：`ChildcareHelper.SuckleFromLactatingPawn`

- `toDrainPool` 按需求与流速上限；食物需求用 `wanted / nutritionPerPoolUnit`（1f），能量需求用 `wanted * nutritionToEnergyFactor / nutritionPerPoolUnit`。
- `nutritionEquivalent = actualDrained * nutritionPerPoolUnit`；食物：`CurLevel += nutritionEquivalent`；能量：`CurLevel += nutritionEquivalent / nutritionToEnergyFactor`（池单位→营养→能量换算一致）。

**结论**：吸奶/消费侧 1:1 与设置一致。

---

## 3. 吸收延迟与药物进水

**位置**：`WorldComponent_MilkCumAbsorptionDelay.ApplyDelayedLactating`

- 若有吸收延迟 Hediff 则先移除；**无论是否有该 Hediff**，均计算 `deltaS = rawSeverity * GetProlactinToleranceFactor(toleranceBefore) * GetRaceDrugDeltaSMultiplier(pawn)` 并执行 `GetOrAddHediff(Lactating)` + `AddFromDrug(deltaS)`。
- 原写法中 `if` 仅控制 `RemoveHediff`，注释与 `deltaS` 缩进易被误读为属于 if；已改为将 `deltaS` 与后续逻辑明确置于方法级，避免误解。

**结论**：逻辑正确；缩进与注释已修正，审阅时已修复。

---

## 4. 已检查的边界与常量

| 项 | 说明 |
|----|------|
| `ExtraNutritionPerDay()` | `flow * NutritionPerPoolUnit`，与 Need_Food 扣减来源一致 |
| `GetReabsorbedNutritionPerDay()` | 回缩池量×步频→日池量，再×NutritionPerPoolUnit×效率，与 Need_Food 加回一致 |
| `lactationExtraNutritionBasis` | 0 不扣/不加，300 为 2 倍，150 为 1:1，行为符合预期 |

## 6. 乳池增加与营养扣除是否同步？

**结论：已改为同一时刻、先扣营养再加池；满池回缩为先回缩再加回营养；吸奶本身即同次调用内扣池并给消费者加营养。**

| 事件 | 同步方式 | 位置 |
|------|----------|------|
| **产奶** | 每 60 tick 先扣饱食/能量再进水 | `UpdateMilkPools()` 开头扣 `ExtraNutritionPerDay()*factor*(60/60000)`，随后 `TickGrowth` 进水。 |
| **满池回缩** | 同一 60 tick 内先回缩再加回 | `UpdateMilkPools()` 中 shrink 循环内累加 `reabsorbedPoolThisStep`，循环后立刻按该量×效率×factor 加回 `Need_Food` 或 `Need_Energy`。 |
| **吸奶** | 同一次调用内完成 | `ChildcareHelper.SuckleFromLactatingPawn` / `GreedyConsume_Prefix`：先 `DrainForConsume` 扣池，同一方法内给婴儿/消费者 `CurLevel +=` 营养或能量等价，无需改动。 |

`Need_Food.NeedInterval` 补丁不再对泌乳 pawn 做任何扣减或加回（均已并入 UpdateMilkPools）。

---

## 7. 总结

- **营养↔乳池 1:1** 在产奶扣饱食、回缩加饱食、吸奶/Charge 消费三条路径上一致。
- **吸收延迟** 逻辑正确；`ApplyDelayedLactating` 缩进与注释已修正。
- **乳池增加与营养扣除**：产奶在 `UpdateMilkPools` 内同一 60 tick **先扣营养再加池**；满池回缩在同一方法内**先回缩再加回营养**；吸奶在 `SuckleFromLactatingPawn`/`GreedyConsume` 同次调用内扣池并给消费者加营养。`NeedInterval` 补丁不再处理泌乳相关扣/加。
- 未发现与规格或 1:1 设计冲突的逻辑错误；仅有一处注释乱码建议清理。
