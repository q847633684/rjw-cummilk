# 乳汁（母乳）系统内子系统划分与目录建议

- **前提**：乳汁 = 母乳/奶，对应目录 `Fluids/Lactation/`。其内包含**泌乳系统**、**池系统**、**挤奶/吸奶系统**三个逻辑子系统。
- **目的**：明确三个子系统的职责边界，并给出「目录怎么分」的可行方案（文档标注 vs 实际拆目录）。

---

## 一、三个子系统的职责划分

| 子系统 | 职责 | 当前主要落点（文件/类型） |
|--------|------|----------------------------|
| **泌乳系统** | L（泌乳量）、L 衰减、药物/分娩进水、Lactating hediff、吸收延迟、耐受、炎症、喷乳反射 R 的存储与衰减 | `Hediffs/`（HediffWithComps_MilkCumLactating、Hediff_ProlactinTolerance、Hediff_LactatingGain、HediffComp_AbsorptionDelayTip）、`World/`（WorldComponent_MilkCumAbsorptionDelay） |
| **池系统** | 双池状态（左/右满度）、容量、进水（L→池，TickGrowth）、压力因子、R 对流速的倍率、选侧用到的池条目与 key | `Comps/CompEquallyMilkable`（池状态 + TickGrowth + 满度/容量）、`Fluids/Shared/` 的 LactationPoolState、BreastPoolEntry、`Helpers/PawnMilkPoolExtensions`（GetBreastPoolEntries、容量/流速）、`Data/` 中池相关（Breastfeed、RaceMilkType 容量系数等） |
| **挤奶/吸奶系统** | 取奶（DrainForConsume）、选侧规则、挤奶/哺乳 Job、WorkGiver、哺乳菜单与权限 | `Comps/CompEquallyMilkable.DrainForConsume`、`Givers/`（WorkGiver_MilkCumMilk、JobGiver_*Breastfeed*、JobGiver_ActiveSuckle 等）、`Jobs/`（JobDriver_MilkCumMilk、JobDriver_ActiveSuckle 等）、`Helpers/ChildcareHelper`、`Helpers/MenuHelper`，UI 中 FloatMenu 哺乳入口 |

**说明**：`CompEquallyMilkable` 同时承担「池状态 + 进水」和「DrainForConsume」，因此横跨池系统与挤奶/吸奶系统；若不做 Comp 拆分类重构，该文件在目录上只能归属其一（建议归池系统，因池是数据主体）。

---

## 二、目录划分方案

### 方案 A：仅文档标注（推荐，零改造成本）

**做法**：不移动任何文件，保持现有 `Lactation/Comps`、`Lactation/Hediffs`、`Lactation/Data`、`Lactation/Helpers`、`Lactation/Jobs`、`Lactation/Givers`、`Lactation/Thoughts`、`Lactation/World` 结构。在记忆库与 Source 索引中写明「逻辑归属」：

- **泌乳系统** ↔ Hediffs、World
- **池系统** ↔ Comps（CompEquallyMilkable 等）、Fluids/Shared 池类型、Helpers 中池/流速相关（如 PawnMilkPoolExtensions）、Data 中池相关
- **挤奶/吸奶系统** ↔ CompEquallyMilkable.DrainForConsume、Givers、Jobs、Helpers 中 Childcare/Menu、UI FloatMenu

**优点**：无需改命名空间、引用、csproj；新人通过文档即可按「泌乳 / 池 / 挤奶吸奶」三个概念找代码。  
**缺点**：目录树不能「一眼看出」三个子系统，只能靠文档与索引。

---

### 方案 B：Lactation 下按子系统建子目录（实际拆目录）【已选用】

**做法**：在 `Fluids/Lactation/` 下再建一层，一个子系统一个目录，再把现有文件按职责迁入。

**补充：除泌乳 / 池 / 挤奶吸奶外，需要单独顾及的模块**

| 模块 | 说明 | 归属 |
|------|------|------|
| **数据（Data）** | RaceMilkType、MilkTag、Breastfeed、MilkSettings、Gene_MilkTypeData、AsyncCollection 等，多系统共用 | 保留在 `Lactation/Data/` 根下，作为乳汁内共享数据，三个子系统都引用 |
| **思想（Thoughts）** | 成瘾满足、催乳素戒断、高耐受低奶、满池、长期未挤奶、长期药物泌乳等 | 按内容拆入：泌乳 → Lactation/Thoughts；池 → Pool/Thoughts（满池）；挤奶吸奶 → Milking/Thoughts（长期未挤奶） |
| **权限 / 设置** | MilkPermissionExtensions、MilkSettings（谁可挤奶/被喂/哺乳） | 归入**挤奶/吸奶系统**（Milking/Helpers、Milking 使用的 Data） |
| **健康 / 乳腺炎** | MilkRelatedHealthHelper（乳腺炎 MTB、躯干/乳房受伤） | 归入**泌乳系统**（Lactation/Helpers），与炎症 I、hediff 相关 |
| **药物摄入** | JobDriver_InjectLactatingDrug、吸收延迟 WorldComponent | 归入**泌乳系统**（Lactation/Jobs、Lactation/World） |
| **精液桶链接** | CompCumBucketLink、CompProperties_CumBucketLink（产奶→精液桶） | 归入**池系统**（Pool/Comps），池产出与桶对接 |
| **通用基础设施** | ListerAsync、ExtensionHelper、Categories、CompProperties_ShowProducer 等 | 保留在 `Lactation/Common/`（或 `Shared/`），供三子系统共用 |

**命名空间约定**（迁移后需全局替换并更新 Harmony/UI/Integration/Core 的 using 与类型引用）：

- 泌乳：`MilkCum.Fluids.Lactation.Lactation.Hediffs`、`MilkCum.Fluids.Lactation.Lactation.World`、`MilkCum.Fluids.Lactation.Lactation.Jobs`、`MilkCum.Fluids.Lactation.Lactation.Thoughts`、`MilkCum.Fluids.Lactation.Lactation.Helpers`
- 池：`MilkCum.Fluids.Lactation.Pool.Comps`、`MilkCum.Fluids.Lactation.Pool.Helpers`、`MilkCum.Fluids.Lactation.Pool.Thoughts`
- 挤奶吸奶：`MilkCum.Fluids.Lactation.Milking.Givers`、`MilkCum.Fluids.Lactation.Milking.Jobs`、`MilkCum.Fluids.Lactation.Milking.Thoughts`、`MilkCum.Fluids.Lactation.Milking.Helpers`
- 数据：`MilkCum.Fluids.Lactation.Data`（保持不变，仍在 Lactation 根下）
- 通用：`MilkCum.Fluids.Lactation.Common`

**需要处理的点**：

- **CompEquallyMilkable**：仍放在 Pool/Comps，Milking 的 Givers/Jobs 通过引用该 Comp 调用 DrainForConsume。
- **Shared 池类型**：LactationPoolState、BreastPoolEntry 若继续放在 `Fluids/Shared/`，则池系统只「使用」Shared；若希望池系统自洽，可迁入 `Lactation/Pool/Data` 并改命名空间（影响面大）。
- **执行**：实际迁移需修改各文件 namespace、csproj 包含路径、以及 Harmony/UI/Integration/Core 中对 Lactation 类型的所有引用，建议分步执行并每步编译通过后再继续。

**优点**：目录与「泌乳 / 池 / 挤奶吸奶」及 Data、Common 一一对应，便于按子系统浏览与扩展。  
**缺点**：需要移动大量文件、改命名空间与引用，测试与回归成本高。

---

### 方案 C：折中（仅新增一层“逻辑分组”目录，部分迁移）

**做法**：保留现有 Comps、Hediffs、Data、Helpers 等**名称**，但在其上层增加三个“逻辑组”目录，只做**部分**迁移，减少改动量：

- `Lactation/Driving/`（泌乳驱动）→ 仅放 Hediffs、World，命名空间如 `MilkCum.Fluids.Lactation.Driving.Hediffs`。
- `Lactation/Pool/`（池）→ 仅放 CompEquallyMilkable 及池相关 Data/Helpers（PawnMilkPoolExtensions 等）。
- `Lactation/Consume/`（挤奶/吸奶）→ 仅放 Givers、Jobs、ChildcareHelper、MenuHelper。

Data、Thoughts 等仍可留在 `Lactation/` 根下共用，避免为每个子系统各复制一份 Data。

**优点**：目录能体现三个子系统，迁移量小于方案 B。  
**缺点**：仍要改命名空间与引用；根下若保留 Data/Thoughts，则「一系统一目录」在乳汁内部不完全严格。

---

## 三、建议结论

- **已选用方案 B**：按上文「方案 B 完整目录与文件映射」执行迁移；Data 与 Common 保留在 Lactation 根下，Thoughts 按内容拆入 Lactation/Pool/Milking。
- 实际迁移需修改各文件 namespace、csproj、以及 Harmony/UI/Integration/Core 中对 Lactation 类型的所有引用，建议分步执行（如先迁 Lactation/，再 Pool/，再 Milking/，最后 Data 与 Common），每步编译通过后再继续。
- 迁移完成后请同步更新 [Source 目录结构梳理](Source目录结构梳理.md) 与 Source/索引.md。
