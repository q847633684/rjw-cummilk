# 记忆库建议与改进

本文档记录对当前记忆库的**改进建议**，便于后续逐步落实。不强制一次性做完，按需采纳。

---

## 一、维护与保鲜

| 建议 | 说明 |
|------|------|
| **条目加「最后核对」** | 在重要条目末尾加一行「最后与代码核对：YYYY-MM 或 commit」。大改后顺手更新，避免记忆与代码脱节。 |
| **改代码时触发更新** | 修改与 L/双池/耐受/进水/衰减/选侧/吸收延迟相关的逻辑时，顺带检查并更新 记忆库 中对应 design/decisions/domain 条目；规则 `use-memory-bank.mdc` 已要求「新增重要结论时写入」，可补充「修改相关逻辑时检查并更新既有条目」。 |
| **废弃与替代** | 某条决策被推翻时，在原文顶部注明「**已废弃**，由新条目（相对路径）替代」，并在 index 的该行加「(已废弃)」。避免 AI 继续引用旧结论。 |

---

## 二、覆盖补全

| 建议 | 说明 |
|------|------|
| **参数联动** | 若存在《参数联动表》或高级设置中参数对行为的影响说明，可在 `domain/` 或 `conventions/` 增加一条「设置参数与行为联动」摘要，并链接到 Docs 对应章节。 |
| **兼容与边界** | 与 RJW/原版/其它 mod 的兼容要点、已知边界情况（如无 RJW 乳房时退化为单对）、存档迁移注意点，可放在 `design/` 或单独 `compat/` 摘要，便于排查问题。 |
| **快速事实** | 零散但常被问的「小事实」（如某 Def 的 defName、某常量数值）可集中在 `domain/快速参考.md` 或沿用 index 的按标签跳转，避免为每一条都建单独文件。 |

---

## 三、检索与使用

| 建议 | 说明 |
|------|------|
| **入口再突出** | 在 README 或项目根 RULE/AGENTS 中写一句：「涉及泌乳/双池/耐受/设置时，先看 记忆库/index.md 和 记忆库/domain/00-概览」。保证人类与 AI 都优先想到记忆库。 |
| **标签表** | 在 index 或 schema 中维护一份「推荐标签表」（如 #双池 #PairIndex #选侧 #耐受 #EMDefOf #tick #吸收延迟 #兼容），新增条目时优先从表里选，便于按标签聚合。 |
| **代码内引用** | 关键逻辑的注释中显式写「见 记忆库/design/双池与PairIndex」或「见 记忆库/decisions/ADR-001」，便于从代码反查到记忆。 |

---

## 四、结构与流程

| 建议 | 说明 |
|------|------|
| **版本/适用范围** | 若项目有多版本分支（如 1.4/1.6），可在架构说明或 README 注明「当前记忆库默认针对 1.6」；某条仅适用于某版本时在条目标题或文首注明。 |
| **何时写记忆** | 在 README「维护」或各 00-概览 中加一句：「何时写：实现或重构做出非显然选择时；与规格/Docs 的对应关系需要固定时；某次审查/讨论得出可复用结论时。」减少「不知道该不该写」的犹豫。 |
| **不追求大而全** | 记忆库以摘要与索引为主，不必把 Docs 全文搬进来；新条目优先写「结论 + 指向 Docs/代码」，保持单条简短可扫读。 |

---

## 五、可选进阶

- **变更日志**：若希望可追溯「记忆库本身」的变更，可在记忆库根目录维护 `changelog-memory.md`，每次新增/废弃/重要修订时记一行（日期 + 条目 + 变更类型）。
- **与待办联动**：若《四层模型升级-代码待办》等 Doc 中有与设计决策强相关的待办，可在对应 design/decision 条目里加「待办：见 Docs/xxx」，便于从记忆反推到任务。

以上可按优先级分批做：先做「废弃与替代」「何时写记忆」「代码内引用」和「改代码时检查记忆」，再视需要补参数联动、兼容摘要和快速参考。

---

## 六、整理时做哪些事（定期整理 checklist）

每次做「记忆库整理」时，建议按下列项过一遍，避免只做索引补全而漏掉其它。

| 项 | 内容 |
|----|------|
| **1 盘点与补全** | 列出各子目录下全部文件；索引/README 是否缺链、错链；分类是否合理并补全说明。 |
| **2 分类与命名** | 命名约定是否写清（见 README「命名与收口约定」）；新文件名是否符合 design/decisions/docs/domain/conventions 的约定。 |
| **3 入口与导航** | 总索引 index、docs/README、各 00-概览 是否有一致入口；按标签/主题的快速跳转是否覆盖常用项。 |
| **4 去重与收口** | 是否有重复或冗余内容（如同一主题多文件且未注明互补关系）；收口规则是否遵守：规格仅 docs/、ADR 仅 decisions/、引用统一用 记忆库/docs/ 或 docs/。 |
| **5 约定与保鲜** | README 维护与命名/收口约定是否更新；关键条目是否加了「最后与代码核对」或「最后整理：YYYY-MM」。 |
| **6 清理与优化** | 是否标出已废弃条目并在 index 注「(已废弃)」；链接与标题格式是否统一；必要时合并或拆分过长/过碎条目。 |

整理完成后在 `changelog-memory.md` 记一笔（日期 + 整理范围 + 主要变更）。
